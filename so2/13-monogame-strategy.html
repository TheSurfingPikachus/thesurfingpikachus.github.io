MonoGame – Strategy Pattern

Situering

We gaan verder met de code van de vorige les en gaan eens kijken naar het bewegingsgedrag van de haai. Tot nu toe zit het gedrag van de haai (de beweging) rechtstreeks in de klasse zelf.

Je hebt iets zoals:

public class Shark : Sprite
{
    ...

    public override void Update(GameTime gameTime)
    {
        Position.Y += Speed;
    }
}

Dit werkt, maar heeft een belangrijk nadeel:

Het gedrag van de haai zit vast in de klasse zelf.

Wat als we variatie willen?

  • sommige haaien bewegen recht naar links

  • andere bewegen zigzag

  • andere bewegen sneller of trager

Dan moeten we de klasse aanpassen of dupliceren.

Zonder Strategy Pattern

Een typische (maar minder goede) oplossing is:

public enum SharkMovementType
{
    Straight,
    Diagonal
}
public class Shark : Sprite
{
    public float Speed { get; set; }
    public SharkMovementType MovementType { get; set; }

    ...

    public override void Update(GameTime gameTime)
    {
        if (MovementType == SharkMovementType.Straight)
        {
            Position.Y += Speed;
        }
        else if (MovementType == SharkMovementType.Diagonal)
        {
            Position.X -= Speed;
            Position.Y += 1;
        }
    }
}

Problemen:

  • Shark krijgt meerdere verantwoordelijkheden

  • if/else groeit naarmate er meer types bijkomen

  • elke nieuwe beweging vereist aanpassing van de klasse

  • moeilijk uitbreidbaar

Strategy Pattern

In plaats van het gedrag in de klasse zelf te stoppen:

We halen het gedrag eruit en stoppen het in aparte objecten.

De haai krijgt dan een strategie die bepaalt hoe hij beweegt door middel van het toepassen van het Strategy Pattern.

Stap 1 – Interface

public interface ISharkMovement
{
    void Update(Shark shark);
}

Deze interface zegt:

Elke beweging moet weten hoe ze een haai moet updaten.

Stap 2 – Concrete strategieën

Rechte beweging

public class StraightMovementStrategy : IMovementStrategy
{
    public void Update(Shark shark)
    {
        shark.UpdatePositionX( - shark.Speed);
    }
}

Diagonal beweging

public class DiagonalMovementStrategy : IMovementStrategy
{
    public void Update(Shark shark)
    {
        shark.UpdatePosition(-shark.Speed, shark.Speed * 0.25F);
    }
}

Stap 3 – Integratie in Shark

public class Shark : Sprite
{
    private IMovementStrategy _movementStrategy;

    public Shark(Texture2D texture,
                 Vector2 position,
                 IMovementStrategy movementStrategy)
        : base(texture, position)
    {
        _movementStrategy = movementStrategy;
    }

    public override void Update(GameTime gameTime)
    {
        _movementStrategy.Update(this);
    }
}

De haai weet nu niet meer hoe hij beweegt.

Hij delegeert dat aan de strategie.

Stap 4 – Gebruik via Factory

public Shark Create(Texture2D texture, float x, float y)
{
    // Typisch zou je hier de movementStrategy meekrijgen, maar om het voorbeeld eenvoudig te houden maken we een
    // willekeurig getal
    IMovementStrategy movementStrategy;

    if (Random.Shared.Next(2) == 0)
    {
        movementStrategy = new StraightMovement();
    }
    else
    {
        movementStrategy = new DiagonalMovement();
    }

    return new Shark(texture, new Vector2(x, y), movementStrategy);
}

Nu kunnen we eenvoudig variëren:

  • verschillende snelheden

  • verschillende bewegingen

  • combinaties

zonder de Shark klasse aan te passen.

Strategy vs State – waarom hier Strategy?

Op het eerste zicht lijkt dit probleem ook oplosbaar met een state machine.

We zouden kunnen zeggen:

  • de haai zit in StraightState

  • of in DiagonalState

Waarom is dit dan géén goede State case?

Het verschil zit niet in de code, maar in de betekenis.

Wat is een state?

Een state stelt een toestand van een object voor.

Typisch:

  • een vijand die idle → chase → attack gaat

  • een speler die springt → valt → landt

  • een game die menu → play → pause gaat

Belangrijk:

States veranderen tijdens de levensduur van het object.

Er is meestal:

  • overgangslogica

  • afhankelijkheid van input of events

Wat gebeurt er bij onze haai?

Bij de haai:

  • hij is gewoon een haai

  • hij blijft een haai

  • zijn gedrag verandert niet tijdens runtime - dit kan in principe wel, maar is niet de focus

We kiezen enkel bij creatie:

  • deze haai beweegt recht

  • deze haai beweegt diagonal

Er zijn:

  • geen transitions

  • geen events die gedrag veranderen

  • geen “toestanden” in de betekenis van een lifecycle

Wanneer zou State wél logisch zijn?

Als we dit zouden hebben:

  • haai start met rechte beweging

  • bij bepaalde conditie → diagonal

  • bij andere conditie → versnellen

  • bij collision → andere reactie

Dan krijgen we:

  • transitions

  • lifecycle

  • toestand van het object

Dan wordt State wél logisch.

Samenvatting

Kernidee:

De haai weet dat hij beweegt, maar niet hoe hij beweegt.

Dit is een typisch voorbeeld van het Strategy Pattern:

  • gedrag wordt losgekoppeld van de klasse

  • gedrag wordt uitwisselbaar

  • de juiste strategie wordt gekozen bij creatie

Belangrijk onderscheid:

State = verandering van toestand Strategy = keuze van gedrag

In deze les gebruiken we Strategy omdat we variatie in gedrag modelleren, niet een toestand van het object.
