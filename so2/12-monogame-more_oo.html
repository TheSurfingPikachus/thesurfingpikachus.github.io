Object Oriënted Programming

We vertrekken vanuit de code van vorige week. In de vorige les hebben we een state machine geïntroduceerd:

  • MenuState

  • PlayState

  • PauseState

  • GameOverState

Elke state bevat zijn eigen gedrag (Update en Draw). Game1 delegeert enkel nog naar de actieve state.

Hoewel dit al een duidelijke verbetering was, zit er nog steeds veel logica rechtstreeks in MonoGame-objecten:

  • Keyboard.GetState()

  • Content.Load

  • SpriteBatch.Draw

  • Vector2 creatie overal in de code

  • constanten verspreid doorheen de code

In deze les gaan we de code verder objectgeoriënteerd structureren.

We willen:

  • verantwoordelijkheden beter scheiden

  • MonoGame afhankelijkheden isoleren

  • objecten modelleren

  • objectcreatie centraliseren

  • configuratie centraliseren

We introduceren hiervoor een aantal nieuwe concepten, alsook houden we in gedachten dat we de MonoGame-objecten zoveel mogelijk willen verbergen achter onze eigen klassen om onze code los te koppelen van het framework.

De code hieronder toont één mogelijke implementatie van het ontwerp dat we bespreken.

Neem deze dus niet zomaar over. Denk na over de gemaakte keuzes en hoe ze passen binnen de architectuur.

In softwareontwerp bestaan er vaak meerdere correcte oplossingen.

Input detecteren

MonoGame gebruikt:

Keyboard.GetState().IsKeyDown(Keys.P)

Het probleem met deze code is dat zolang een toets ingedrukt blijft, blijft deze true.

Voor veel gameplay acties willen we echter de actie slechts één keer uitvoeren wanneer de toets net werd ingedrukt.

Voorbeeld:

  • menu starten

  • pauze togglen

  • schieten

  • interactie

Als we de vorige keyboard state bijhouden, dan kunnen we deze vergelijken met de huidige state. Op deze manier weten we of een toets nieuw ingedrukt is.

private KeyboardState _previous;
private KeyboardState _current;

Elke frame (de update method):

_previous = _current;
_current = Keyboard.GetState();

Daarna kunnen wij deze states bevragen en weten of de toets nieuw ingedrukt is:

public bool WasKeyJustPressed(Keys key)
{
    return _current.IsKeyDown(key) &&
           !_previous.IsKeyDown(key);
}

Dit is een typisch probleem in MonoGame.

We gaan dit echter niet rechtstreeks in onze game code plaatsen, maar isoleren via het Facade Pattern.

KeyboardFacade

We willen MonoGame code scheiden van onze eigen code.

Daarom introduceren we:

public static class KeyboardFacade
{
    private static KeyboardState _previous;
    private static KeyboardState _current;

    public static void Update()
    {
        _previous = _current;
        _current = Keyboard.GetState();
    }

    public static bool IsKeyDown(Keys key)
        => _current.IsKeyDown(key);

    public static bool WasKeyJustPressed(Keys key)
        => _current.IsKeyDown(key)
           && !_previous.IsKeyDown(key);
}

Voordelen:

  • MonoGame afhankelijkheid zit op één plaats

  • testbaarheid wordt beter

  • game code wordt eenvoudiger

Gebruik:

KeyboardFacade.WasKeyJustPressed(Keys.Space)

GraphicsFacade

In de standaard MonoGame template bevat Game1 een variabele:

GraphicsDeviceManager _graphics;

Dit object beheert onder andere:

  • resolutie

  • fullscreen

  • backbuffer grootte

  • toepassen van grafische wijzigingen

In veel MonoGame projecten wordt deze variabele rechtstreeks gebruikt vanuit Game1. Wanneer andere klassen grafische instellingen nodig hebben, moeten ze vaak opnieuw via Game1 gaan.

Dat creëert twee problemen:

  • veel code wordt afhankelijk van Game1

  • de MonoGame GraphicsDeviceManager lekt doorheen de hele codebase

Daarom introduceren we een GraphicsFacade.

Het idee van een facade is dat we een eenvoudige interface aanbieden naar een complexer systeem. In dit geval maken we voor`GraphicsDeviceManager` een eigen klasse.

Omdat er in een MonoGame game meestal maar één graphics manager bestaat, maken we deze facade static.

Implementatie

public static class GraphicsFacade
{
    private static GraphicsDeviceManager _graphics;

    public static void Initialize(GraphicsDeviceManager graphics)
    {
        _graphics = graphics;
    }

    public static int ScreenWidth
        => _graphics.PreferredBackBufferWidth;

    public static int ScreenHeight
        => _graphics.PreferredBackBufferHeight;

    public static void SetResolution(int width, int height)
    {
        _graphics.PreferredBackBufferWidth = width;
        _graphics.PreferredBackBufferHeight = height;
        _graphics.ApplyChanges();
    }
}

Initialisatie gebeurt één keer in Game1 (dit vervangt de vroegere initializatie van _graphics).

GraphicsFacade.Initialize(_graphics);

Daarna kan de rest van de code grafische informatie opvragen zonder Game1 of GraphicsDeviceManager te kennen.

Voorbeeld:

var screenWidth = GraphicsFacade.ScreenWidth;
var screenHeight = GraphicsFacade.ScreenHeight;

Of:

GraphicsFacade.SetResolution(1920, 1080);

Voordelen

  • andere klassen hoeven Game1 niet te kennen

  • MonoGame-specifieke code zit op één plaats

  • grafische configuratie kan centraal beheerd worden

  • de code wordt eenvoudiger leesbaar

Context object

In onze code zitten nog heel wat class-level variabelen rechtstreeks in Game1.

Denk bijvoorbeeld aan:

  • de speler

  • de vijanden

  • de achtergrond

  • lijsten met objecten

  • timers

  • settings

  • assets

Dat is op zich niet vreemd: Game1 is vaak de eerste plaats waar al die gegevens terechtkomen. Maar naarmate een game groeit, wordt Game1 dan al snel een soort verzamelbak van alles wat de game nodig heeft.

Dat zorgt voor enkele duidelijke nadelen:

  • Game1 krijgt te veel verantwoordelijkheden

  • states en andere objecten moeten voortdurend via Game1 werken

  • class-level variabelen raken verspreid en moeilijk beheersbaar

  • het wordt onduidelijk welke data echt deel uitmaakt van de actuele spelcontext

Daarom halen we die gedeelde toestand uit Game1 en brengen we die onder in een apart Context object.

De context bevat de gedeelde toestand van de game: de objecten, de services en de instellingen die op dat moment nodig zijn.

Wat hoort in de context?

De context is de plaats waar we zaken bijhouden zoals:

  • Player

  • Background

  • collectie van Shark objecten

  • eventueel andere game objecten

  • AssetsManager

  • GameSettings

Met andere woorden: alles wat verschillende delen van de game nodig kunnen hebben en wat vroeger als class-level variabele in Game1 stond.

Voorbeeld

public class GameContext
{
    public Player Player { get; set; }
    public Background Background { get; set; }

    public List<Shark> Sharks { get; } = new();

    ...

    public AssetsManager Assets { get; }
    public GameSettings Settings { get; }

    // Een facade kan ook niet-statisch zijn. Deze kan je dan via bv. Context object ter beschikking stellen
    // public KeyboardFacade Keyboard { get; }

    public GameContext(
        AssetsManager assets, // Dit kan ook static zijn
        //KeyboardFacade keyboard,
        GameSettings settings)
    {
        Assets = assets;
        Settings = settings;
        //Keyboard = keyboard;
    }
}

Omdat GraphicsFacade & KeyboardFacade static is, hoeft die niet noodzakelijk als instantie in de context te zitten. Die kan rechtstreeks statisch gebruikt worden, maar een facade hoeft niet altijd static te zijn. Conceptueel hoort die wel bij dezelfde laag van gedeelde infrastructuur.

Gebruik in Game1

In plaats van losse velden zoals:

private Player _player;
private Background _background;
private List<Shark> _enemies;

krijgen we:

private GameContext _context;

Tijdens de initialisatie vullen we die context op.

_context = new GameContext(
    assetsManager,
    gameSettings);

_context.Player = PlayerFactory.Create(_playerTexture, 100, 300);
_context.Background = BackgroundFactory.Create(_backgroundTexture, 0, 0, BACKGROUND_SCALE);
_context.Sharks.Add(SharkFactory.Create(_sharkTexture, 400, 100, SHARK_SCALE));

Waarom is dit nuttig?

Door de gedeelde toestand te bundelen in een context:

  • wordt Game1 kleiner en overzichtelijker

  • weten we duidelijk welke data tot de actuele gamecontext behoort

  • kunnen states en andere objecten werken met één context in plaats van met allerlei losse velden

  • wordt de code eenvoudiger uitbreidbaar

In plaats van allerlei losse parameters door te geven zoals:

new PlayState(player, sharks, background, ...)

kunnen we gewoon werken met:

new PlayState(context)

Ontwerpmatig voordeel

Het context object is geen “magische oplossing”, maar wel een nuttige stap om orde te brengen in gedeelde toestand.

Het centrale idee is:

Wat vroeger als class-level variabelen verspreid in Game1 zat, brengen we samen in één object dat de actuele spelwereld en infrastructuur voorstelt.

Zo wordt Game1 minder een opslagplaats van alles, en meer de plaats waar de game-loop draait en de context beheerd wordt.

GameSettings

In veel games zitten constanten verspreid doorheen de code:

  • snelheden

  • spawn tijden

  • schaal

  • schermlimieten

Dit maakt de code moeilijk aanpasbaar.

Daarom centraliseren we alle constanten.

public class GameSettings
{
    public const float PlayerSpeed = 200f;
    public const float SharkSpeed = 150f;

    public const int SharkSpawnInterval = 2000;

    public const float PlayerScale = 1f;
    public const float SharkScale = 1f;
}

Voordelen:

  • configuratie op één plaats

  • eenvoudiger balancen van gameplay

  • duidelijk onderscheid tussen code en configuratie

Dit hoeven geen constanten te zijn, dit kan ook aanpasbare eigenschappen zijn binnen de GameSettings zodat deze bv. uit een bestand komen, of aanpasbaar zijn tijdens het spel.

AssetsManager

MonoGame gebruikt:

Content.Load<Texture2D>("player")

Problemen:

  • strings overal in code

  • assets kunnen meerdere keren geladen worden

  • refactoring is moeilijk

We maken een AssetsManager.

public class AssetsManager
{
    private ContentManager _content;

    private Dictionary<string, Texture2D> _textures
        = new();

    public AssetsManager(ContentManager content)
    {
        _content = content;
    }

    public Texture2D GetTexture(string name)
    {
        if (!_textures.ContainsKey(name))
        {
            _textures[name] =
                _content.Load<Texture2D>(name);
        }

        return _textures[name];
    }
}

Assets namen worden constanten.

public static class AssetNames
{
    public const string Player = "player";
    public const string Shark = "shark";
    public const string Background = "background";
}

Voordelen:

  • lazy loading

  • caching

  • minder string fouten

Gebruik:

var tex = _context.Assets.GetTexture(AssetNames.Player);
Kan ook als static klasse, hangt van de context af. Bij voorkeur proberen we wel statische klassen te vermijden

Sprite basis klasse

Momenteel worden objecten vaak voorgesteld door losse data:

  • texture

  • positie

  • schaal

  • snelheid

Dit is niet objectgeoriënteerd (en komt al rap slordig en onoverzichtelijk over)

We introduceren een basis Sprite klasse.

public class Sprite
{
    protected Texture2D Texture;
    public Vector2 Position { get; set; }

    public float Scale { get; set; }

    public Sprite(Texture2D texture, Vector2 position)
    {
        Texture = texture;
        Position = position;
    }

    public virtual void Update(GameTime gameTime)
    {
    }

    public virtual void Draw(GraphicsFacade graphics)
    {
        graphics.Draw(Texture, Position);
    }
}

Dit wordt de basis voor alle objecten.

Player object

public class Player : Sprite
{
    private InputService _input;

    public float Speed { get; init; }

    public Player(Texture2D texture,
                  Vector2 position,
                  float speed,
                  InputService input)
        : base(texture, position)
    {
        Speed = speed;
        _input = input;
    }

    public override void Update(GameTime gameTime)
    {
        if (_input.ShouldGoLeft())
            Position.X -= Speed;

        if (_input.ShouldGoRight())
            Position.X += Speed;

        ...
        // We hebben natuurlijk ook nog Up & Down
    }
}

Belangrijk:

De player gebruikt InputService.

InputService

De player moet niet weten hoe input precies werkt.

We maken daarom een InputService.

public class InputService
{
    // We hoeven de KeyboardFacade niet altijd als static te implementeren, je kan deze ook niet statisch maken en meegeven aan de InputService
    private KeyboardFacade _keyboard;

    public InputService(KeyboardFacade keyboard)
    {
        _keyboard = keyboard;
    }

    public bool ShouldGoLeft()
        => _keyboard.IsKeyDown(Keys.Left);

    public bool ShouldGoRight()
        => _keyboard.IsKeyDown(Keys.Right);

    ... // Andere methods
}

Voordelen:

  • player kent keyboard niet

  • input kan later vervangen worden

  • duidelijke mapping van acties

Factory pattern

Object creatie kan complex worden.

Bijvoorbeeld:

new Shark(texture, new Vector2(x,y))
{
    Speed = 150f,
    Scale = 1f
};

Wanneer dit op meerdere plaatsen gebeurt, krijgen we duplicatie.

Daarom gebruiken we een Factory Pattern.

Factories kunnen gemaakt worden per object type, of per domein. We kunnen een SharkFactory hebben, maar we zouden evengoed kunnen groeperen en een SpriteFactory hebben.

SharkFactory

public class SharkFactory
{
    private Context _context;

    public SharkFactory(Context context)
    {
        _context = context;
    }

    public Shark Create(float x, float y)
    {
        var texture =
            _context.Assets.GetTexture(
                AssetNames.Shark);

        var shark =
            new Shark(texture, new Vector2(x, y));

        shark.Speed =
            GameSettings.SharkSpeed;

        shark.Scale =
            GameSettings.SharkScale;

        return shark;
    }
}

Voordelen:

  • objectcreatie gecentraliseerd

  • minder duplicatie

  • zeer gemakkelijk om variaties te maken

Bijvoorbeeld:

  • CreateFastShark

  • CreateBossShark

  • CreateRandomShark

Samenvatting

We hebben een aantal belangrijke OO concepten toegepast:

  • Inheritance

  • Sprite → Player

  • Sprite → Shark

  • Sprite → Background

  • Facade pattern

  • GraphicsFacade

  • KeyboardFacade

  • Factory pattern

  • SharkFactory

  • PlayerFactory

  • Service pattern

  • InputService

  • Context object

  • bundelt alle globale services

  • Assets manager

  • centraliseert asset loading

  • GameSettings

  • centraliseert configuratie

Deze aanpak maakt de game:

  • beter uitbreidbaar

  • beter onderhoudbaar

  • minder afhankelijk van MonoGame

  • duidelijker qua verantwoordelijkheden
