MonoGame – van if/else naar een statemachine

Situering

In deze les bouwen we verder op het MonoGame-project uit de vorige les. In deze versie zit de speltoestand rechtstreeks in Game1 via een enumeratie en een veld om de huidige state bij te houden. De Update- en Draw-methode bevatten dan grote if/else if-blokken die bepalen wat er moet gebeuren in Menu, Play, Pause en GameOver. In de nieuwe versie wordt dat vervangen door een statemachine met een abstracte basisklasse AbstractState en concrete toestanden zoals MenuState, PlayState, PauseState en GameOverState. Game1 houdt dan enkel nog een referentie bij naar de actieve state en delegeert Update en Draw volledig naar die state.

Startpunt: alles zit in Game1

In Update wordt met een reeks if-constructies gekeken welke logica uitgevoerd moet worden. In Draw gebeurt hetzelfde voor de rendering. Daardoor krijgt Game1 meerdere verantwoordelijkheden: het beheert de game-loop, bewaart gedeelde data, verwerkt input, voert spelregels uit en beslist ook nog eens welke toestand actief is.

private States _currentState;
if (_currentState == States.Menu)
{
    ...
}
else if (_currentState == States.Play)
{
    ...
}
else if (_currentState == States.Pause)
{
    ...
}
else if (_currentState == States.GameOver)
{
    ...
}

Probleem van deze aanpak

Voor een klein voorbeeld werkt deze oplossing nog. Maar zodra een spel groeit, duiken er snel problemen op:

  • Game1 wordt te groot;

  • logica van verschillende toestanden raakt door elkaar;

  • het wordt moeilijker om gedrag per toestand apart te begrijpen;

  • nieuwe states toevoegen betekent opnieuw if/else-blokken uitbreiden;

  • hergebruik en testen worden moeilijker.

Concreet zie je in de code dat menu-, speel-, pauze- en gameoverlogica allemaal in dezelfde Update-methode zitten. Ook in Draw staan meerdere branches naast elkaar. Dat is een klassiek signaal dat gedrag beter opgesplitst wordt in aparte objecten.

Wat is een statemachine?

Een statemachine is een model waarbij een systeem zich altijd in exact één toestand bevindt, en waarbij het gedrag afhangt van die huidige toestand. Een overgang naar een andere toestand gebeurt wanneer een bepaalde gebeurtenis optreedt, bijvoorbeeld een toetsindruk of een collision (2 of meer objecten die elkaar raken). In een game is dat erg herkenbaar:

  • in het menu reageert het spel anders dan tijdens het spelen;

  • tijdens het spelen teken je vijanden en verwerk je beweging;

  • in pauze toon je meestal dezelfde wereld, maar zonder normale update-logica;

  • bij game over toon je een eindscherm en laat je de speler opnieuw starten.

Een statemachine zegt dus eigenlijk:

De toestand bepaalt welk gedrag actief is.

Doelarchitectuur

In code verdwijnt de enum uit Game1. In de plaats daarvan bewaart Game1 een veld van het type AbstractState. Daarnaast krijgt Game1 een methode ChangeState(AbstractState newActiveState). Bij het initialiseren houden we de logica _currentState = States.Menu, maar passen we deze aan naar new MenuState(this). Daarna roept Game1 in zijn game-loop gewoon _currentState.Update(gameTime) en _currentState.Draw(gameTime, _spriteBatch) op.

this hebben we nodig op elke state omdat we de context van ons spel niet in de states willen. Deze halen we - op dit moment - uit de Game1 instantie. 
private AbstractState _currentState;

public void ChangeState(AbstractState newActiveState)
{
    _currentState = newActiveState;
}
protected override void Initialize()
{
    _currentState = new MenuState(this);
    ...
    base.Initialize();
}
protected override void Update(GameTime gameTime)
{
    ...
    _currentState.Update(gameTime);
    ...
    base.Update(gameTime);
}

protected override void Draw(GameTime gameTime)
{
    GraphicsDevice.Clear(Color.CornflowerBlue);
    _spriteBatch.Begin();

    _currentState.Draw(gameTime, _spriteBatch);

    _spriteBatch.End();
    base.Draw(gameTime);
}

De abstracte basisklasse

De kern van de nieuwe oplossing is AbstractState. Deze klasse definieert wat elke state minstens moet kunnen: een Update(…​) en een Draw(…​). Daarnaast bewaart ze een referentie naar de Game1-context, zodat concrete states toegang hebben tot gedeelde data en functionaliteit. De basisklasse bevat ook hulpfuncties om toetsen te lezen.

public abstract class AbstractState
{
    protected Game1 Context { get; init; }

    public AbstractState(Game1 context)
    {
        Context = context;
    }

    public abstract void Update(GameTime gameTime);
    public abstract void Draw(GameTime gameTime, SpriteBatch spriteBatch);

    protected bool IsKeyDown(Keys key1, Keys key2) => IsKeyDown(key1) || IsKeyDown(key2);
    protected bool IsKeyDown(Keys key) => Keyboard.GetState().IsKeyDown(key);
}
We markeren de klasse als abstract omdat we willen afdwingen dat deze NIET kan geïnstantieerd worden.
We maken de methoden Draw en Update abstract omdat we willen afdwingen dat elke state zijn eigen implementatie doet.
We zetten de methoden van IsKeyDown in de abstracte klasse omdat elke state deze nodig heeft

Waarom is dit nuttig?

Deze klasse zorgt voor een gemeenschappelijk contract:

  • elke state moet kunnen updaten;

  • elke state moet kunnen tekenen;

  • elke state krijgt toegang tot dezelfde context;

Dit is ontwerpmatig interessant omdat clientcode dan kan werken tegen de abstractie AbstractState in plaats van tegen concrete types zoals MenuState of PlayState.

De concrete states

MenuState

MenuState is klein en duidelijk: wanneer de speler op Enter drukt, schakelt de game naar PlayState. In Draw wordt enkel de menutekst getoond. Alle logica van het menu zit dus samen in één klasse.

public class MenuState : AbstractState
{
    public MenuState(Game1 context) : base(context) { }

    public override void Update(GameTime gameTime)
    {
        if (IsKeyDown(Keys.Enter))
            Context.ChangeState(new PlayState(Context));
    }

    public override void Draw(GameTime gameTime, SpriteBatch spriteBatch)
    {
        spriteBatch.DrawString(Context.SpriteFont,
            "Druk op enter om te beginnen",
            Vector2.Zero,
            Color.White);
    }
}

PlayState

PlayState bevat de eigenlijke spel-logica. In Update worden achtergrond, speler en vijanden geüpdatet, en wordt eventueel een nieuwe enemy gespawned. Bij P wordt overgeschakeld naar PauseState. In Draw worden achtergrond, enemies en speler getekend. Daardoor zit alle gameplay op één plaats.

public override void Update(GameTime gameTime)
{
    UpdateBackgroundPosition();
    UpdatePlayerPosition();
    UpdateEnemyPositions();
    SpawnEnemyIfNeeded(gameTime);

    if (IsKeyDown(Keys.P))
        Context.ChangeState(new PauseState(Context, this));
}

PauseState

PauseState is hier interessant. Deze state bewaart een referentie naar PreviousState, gezien we enkel willen pauzeren op een PlayState gaan we hier een PlayState binnenpakken en niet AbstractState. In Draw roept ze eerst PreviousState.Draw(…​) op en tekent daarna de pauzetekst erbovenop. Daardoor blijft de wereld zichtbaar, maar ligt de update-logica stil want de PreviousState.Update() wordt niet opgeroepen. Als de speler op Enter drukt, schakelt de game terug naar de vorige state.

public class PauseState : AbstractState
{
    public PlayState PreviousState { get; init; }

    public PauseState(Game1 context, PlayState previousState) : base(context)
    {
        PreviousState = previousState;
    }

    public override void Update(GameTime gameTime)
    {
        if (IsKeyDown(Keys.Enter))
            Context.ChangeState(PreviousState);
    }

    public override void Draw(GameTime gameTime, SpriteBatch spriteBatch)
    {
        PreviousState.Draw(gameTime, spriteBatch);
        spriteBatch.DrawString(Context.SpriteFont,
            "Pause. Druk op enter om door te doen",
            Vector2.Zero,
            Color.White);
    }
}

GameOverState

GameOverState toont een game-over-boodschap en laat de speler met Enter terugkeren naar MenuState. Ook dit gedrag zit volledig apart van de andere toestanden.

Wat verandert er ontwerpmatig?

De overgang van de code zonder naar met statemachine is geen cosmetische refactor. Het ontwerp verandert fundamenteel.

1. Van conditionele logica naar polymorfisme

In de oude code vraagt de code voortdurend: “In welke state zit ik nu?” In de huidige code hoeft Game1 dat niet meer te weten. Het roept gewoon Update en Draw op de huidige state aan. Welke concrete implementatie uitgevoerd wordt, hangt af van het object waar _currentState naar verwijst. Dat is polymorfisme in actie.

2. Betere scheiding van verantwoordelijkheden

Game1 wordt teruggebracht tot zijn rol als centrale game-context:

  • resources laden;

  • gedeelde data bewaren;

  • de actieve state bijhouden;

  • delegeren naar die state.

De concrete states nemen het gedragsaspect over. Dat maakt de code leesbaarder en lokaler: wie het menu wil begrijpen, kijkt naar MenuState; wie gameplay wil begrijpen, kijkt naar PlayState.

3. Uitbreidbaarheid

Een nieuwe toestand toevoegen, zoals bijvoorbeeld SettingsState of HighScoreState, vraagt nu geen uitbreiding van een groot if/else-blok in Game1. Je maakt gewoon een extra klasse die AbstractState erft en je schakelt ernaartoe wanneer nodig. Dat sluit beter aan bij het open/closed-principe: bestaande code hoeft minder aangepast te worden wanneer nieuw gedrag toegevoegd wordt. Dit principe werkt ook als je meerdere PlayStates wil bv. SinglePlayState of MultiPlayState.

Waarom krijgt elke state een Game1-context?

De states krijgen in de constructor een Game1 context. Daardoor kunnen ze werken met gedeelde gegevens zoals:

  • PlayerPosition

  • EnemyPositions

  • BackgroundPosition

  • SpriteFont

  • textures

  • GraphicsDeviceManager

Dat is praktisch, maar ook interessant. De state-objecten beheren niet alles zelf; ze gebruiken de context om met gedeelde speldata te werken. Je kan dit zien als een eenvoudige vorm van het State pattern waarbij Game1 de context is en de concrete states het toestand-afhankelijke gedrag implementeren. De code laat dat patroon duidelijk zien via Context.ChangeState(…​) en de AbstractState-hiërarchie.

Sterktes van deze aanpak

Leesbaarheid

Elke state bevat enkel de code die voor die toestand relevant is. Daardoor wordt de intentie van de code sneller duidelijk.

Onderhoudbaarheid

Aanpassingen in het menu breken minder snel gameplaycode, en omgekeerd. Je werkt in kleinere, meer afgebakende klassen.

Uitbreidbaarheid

Nieuwe states kunnen toegevoegd worden zonder dat Game1 opnieuw een beslisboom moet kennen.

Hergebruik van gedrag

De abstracte basisklasse centraliseert gedeelde input-hulpjes en het gemeenschappelijke contract.
